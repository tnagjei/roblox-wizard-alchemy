# 多语言计划

## 结论

基于 GSC 国家/地区数据，当前站点已经在多个非英语市场获得展示。多语言值得做，但不应该一次性全站翻译。第一阶段应只做高意图页面，尤其是 Codes、Tier List、Value List、Units。

Counter: 不能把国家数据直接等同于语言需求。菲律宾展示量高，但不代表应该马上做 Tagalog。菲律宾 Roblox 用户可能仍以英语搜索为主。

## 对 Gemini 计划的修正

Gemini 的方向有参考价值，但部分数字和 GSC 文件不一致。

需要修正的点：

1. 泰语应进入第一阶段。泰国展示 195，CTR 只有 0.51%，这更像语言和摘要吸引力问题。
2. 西班牙语不应放太后。西语合并市场有展示和点击信号。
3. 葡萄牙语不能只因巴西玩家多就提前。巴西当前 55 展示、0 点击，只能算潜力市场。
4. 俄语区 CTR 没有高到可以第一批优先，但覆盖多国，适合第二阶段。
5. 菲律宾不应直接做 Tagalog。应先用 GSC 查询词验证是否存在本地语言搜索需求。

## 工程实现计划

### 路径策略

采用子目录，不用子域名：

```text
/id/
/th/
/vi/
/zh-tw/
/ko/
/ru/
/es/
/pt-br/
```

原因：

1. 子目录更容易继承主域信号。
2. GSC 管理更简单。
3. Vercel 部署更简单。
4. sitemap、内链、hreflang 更容易维护。

### Next.js 实现建议

当前项目是 JSON 数据驱动站点。第一版不建议直接引入复杂 i18n 框架。

建议使用：

```text
app/[locale]/...
```

新增轻量翻译数据目录：

```text
public/data/i18n/id.json
public/data/i18n/th.json
public/data/i18n/vi.json
public/data/i18n/zh-tw.json
public/data/i18n/ko.json
```

第一阶段只存多语言页面需要的翻译字段，不拆原始英文主数据。

### TDK 和 H1 约束

英文主站：

1. 不改 TDK。
2. 不改 H1。
3. 不改现有主关键词布局。

多语言新页面：

1. 必须有本地化 title。
2. 必须有本地化 description。
3. 必须有本地化 H1。

原因：多语言页面如果仍使用英文 TDK 和 H1，Google 很难把它判断为真正的本地语言页面。

### hreflang

必须为英文和多语言页面建立互相指向关系。

示例：

```html
<link rel="alternate" hreflang="en" href="https://www.noobtowerdefense.org/codes/" />
<link rel="alternate" hreflang="id" href="https://www.noobtowerdefense.org/id/codes/" />
<link rel="alternate" hreflang="th" href="https://www.noobtowerdefense.org/th/codes/" />
<link rel="alternate" hreflang="vi" href="https://www.noobtowerdefense.org/vi/codes/" />
<link rel="alternate" hreflang="zh-TW" href="https://www.noobtowerdefense.org/zh-tw/codes/" />
<link rel="alternate" hreflang="ko" href="https://www.noobtowerdefense.org/ko/codes/" />
<link rel="alternate" hreflang="x-default" href="https://www.noobtowerdefense.org/codes/" />
```

### sitemap

sitemap 应输出：

1. 英文原页面。
2. 多语言页面。
3. 每个 URL 独立 lastModified。
4. 后续支持每个页面独立 changeFrequency 和 priority。

## 内容制作原则

### 不要机器直译

Roblox 攻略站的术语不能机械翻译，尤其是：

1. Codes
2. Tier List
3. Units
4. Value List
5. Scripts
6. Macro
7. Loadout

### 术语处理建议

| 英文词 | 处理建议 |
|---|---|
| Codes | 多数语言保留 Codes，同时解释为兑换码 |
| Tier List | 多数语言保留 Tier List，同时解释为强度排行 |
| Units | 根据语言习惯翻译为单位、角色或塔 |
| Value List | 翻译为价值列表或交易价值表 |
| Scripts | 加安全提醒，不引导外挂 |
| Macro | 保留 Macro，并解释为自动操作配置 |

### 每个语言页保留英文核心词

例如印尼语页面标题可以保留 `Noob Tower Defense Codes`，再补本地语言说明。

原因：玩家经常用英文游戏名和英文功能词搜索。

## 14 天执行计划

### 第 1 到 2 天

1. 固定语言列表：`id`、`th`、`vi`、`zh-tw`、`ko`。
2. 建立 `public/data/i18n/`。
3. 建立 locale 路由。
4. 建立 hreflang 生成函数。
5. sitemap 加入多语言 URL。

### 第 3 到 5 天

先做 Codes 页：

```text
/id/codes/
/th/codes/
/vi/codes/
/zh-tw/codes/
/ko/codes/
```

原因：Codes 页搜索意图最强，也最容易从 GSC 看验证结果。

### 第 6 到 8 天

扩展 Tier List 和 Units：

```text
/id/tier-list/
/id/units/
/th/tier-list/
/vi/tier-list/
/zh-tw/tier-list/
/ko/tier-list/
```

### 第 9 到 10 天

1. 英文页底部加语言切换。
2. 多语言页互链。
3. 首页增加轻量语言入口。
4. 避免让语言入口看起来像机器翻译站。

### 第 11 到 14 天

检查 GSC：

1. 是否被索引。
2. 是否出现本地语言查询。
3. CTR 是否高于英文页。
4. 平均排名是否从 8 到 10 推进到 5 到 7。
5. 哪个语言先出点击。

## 成败判断标准

### 7 天内

只看索引，不看点击。

目标：

```text
多语言 URL 被发现并进入索引
```

### 14 到 21 天

看展示。

目标：

```text
/id/、/th/、/vi/ 至少开始有展示
```

### 30 天

看点击率。

| 语言 | 判断标准 |
|---|---:|
| 印尼语 | CTR > 3.5% |
| 泰语 | CTR > 1.5% |
| 越南语 | CTR > 2% |
| 繁体中文 | CTR > 6% |
| 韩语 | CTR > 5% |

失败标准：

```text
30 天后某语言展示 < 20 且点击 = 0，则不要继续扩该语言。
```

## 风险提醒

1. 多语言不是全站翻译工程。
2. 第一阶段只做少量高意图页面。
3. 不要直接用机器翻译批量铺页面。
4. 不要把国家数据直接当成语言需求。
5. 菲律宾展示高，但暂不建议做 Tagalog。
6. 巴西展示有潜力，但 0 点击说明需要小范围测试。
7. 多语言页必须本地化 TDK 和 H1，否则 SEO 价值会明显下降。
